상속 돌아가기

상속이 무엇인지를 논하기 전에 상속의 필요성에 대해 먼저 생각해보고자 합니다. 회사에서 이런 요구가 주어졌다고 합시다.

"동그라미와, 정삼각형과, 정사각형을 그리는 프로그램을 만들어주세요. 각각 반지름 혹은 한 변의 길이를 인자로 넓이를 구할 수 있어야합니다."

절차적 프로그래밍: 구조체를 만들까요? struct Circle, struct Triangle, struct Rect. 그 후 각 도형별로 넓이를 구할 수 있는 getCircleArea, getTriangleArea, getRectArea라는 메서드를 구현합니다. 이 후 제출합니다.

객체지향적 프로그래밍: 클래스를 만듭니다. class Circle, class Triangle, class Rect. 정답부터 말하자면 class Shape라는 것을 하나 더 만든 후, Circle, Triangle, Rect가 이를 상속하도록 만듭니다. Shape 클래스에 구현된 getArea를 각각 오버라이드하여 각 도형별로 알맞게 사용합니다.

(다음날..)

추가요구 : "아..미안한데, 스펙이 바뀌어서 오각형과 육각형을 추가해야할 것 같아."

절차적 프로그래밍: 이런, 다급히 Pent와 Hex구조체를 만듭니다. getPentArea, getHexArea 메서드를 만듭니다. 코드 여기저기에 else if(도형==오각형) else if(도형==육각형)과 같은 조건문을 덕지 덕지 넣어야합니다. 왠지모르게 잘 돌아갈 것 같은 코드가 꼬이기 시작합니다.

객체지향적 프로그래밍: 클래스를 만듭니다. class Pent, class Hex. 둘 다 class Shape라는 것을 상속받게합니다. getArea를 오버라이드하여 오각형과 육각형의 넓이를 구할 수 있게 해줍니다. Shape별로 알아서 해당 도형의 getArea 메서드를 사용하게 구현되어 있으므로, 다른 코드는 건드릴 필요가 없습니다.

이처럼 상속을 활용하면 코드의 확장의 유연해집니다.

추상 클래스 돌아가기

abstract 명령어를 이용하여, 객체가 직접적으로 선언되는 것이 아니라 상속을 통해서만 사용되도록 유도하는 것이 추상 클래스입니다.

메서드의 앞에 abstract 키워드를 붙이면 이는 추상 메서드가 되며 오버라이드를 강제하게 됩니다.

인터페이스 돌아가기

다중 상속을 허용하지 않는 자바에서, 다중 상속이 불가능하여 해결할 수 없는 문제들을 해결하기 위해 생겨난 것이 인터페이스 입니다. 인터페이스는 모든 메서드를 춧항 메서드로 정의해야하며, 하위 클래스에서는 이를 반드시 구현해야합니다. 얼핏 생각하면 인터페이스 자체에는 전혀 코드를 구현할 수 없으니까 있으나 마나가 아닌가라고 생각할 수도 있지만 이는 오산입니다. 인터페이스를 인자나 리턴 유형으로 사용하게 되면, 이 인터페이스를 구현하는 모든 객체는 여기에 사용될 수 있으므로 큰 의미가 있습니다.

메서드의 앞에 abstract 키워드를 붙이면 이는 추상 메서드가 되며 오버라이드를 강제하게 됩니다.
